EveryDayMeds SCR Pre-Screen

Complete reference — what data is read, what is searched, and how PASS/FAIL works.
Extension v2.0.x · Based on MedExpress keyword rules (R001–R006)

End-to-end flow

1. Order pageScrape name, DOB, sex from Rx order DOM
2. NHS portalAuto search → patient → Clinical tab → SCR HTML
3. Keyword scanSearch entire SCR text for rule keywords

Phase 1 — Order page (before NHS)

When you click Go to NHS SCR, the extension reads from the EveryDayMeds order page only — not from the SCR yet.

Phase 2 — NHS portal automation

This navigates the NHS site; it does not parse clinical content until the Clinical tab loads.

  1. Confirm Smartcard org (EveryDayMeds / FJE59)
  2. Advanced search: gender + first name + surname + single DOB → Find patient
  3. Click matching search result row
  4. Click Clinical tab
  5. Click consent Yes if modal appears
  6. Extract SCR HTML from clinical page

SCR HTML captured from (first match):

  • .scr-print-wrapper innerHTML, or
  • .clinicals-page (if it contains SCR tables/sections), or
  • Area around table.scr-table / “General Practice Summary”

File: automate-scr.js

Phase 3 — Keyword scan logic

How matching works

1. SCR HTML → flatten to one long text string (all tags stripped, spaces normalised)
2. For each rule where fail_if_found: true, check every keyword
3. Match is case-insensitive substring (e.g. “depress” matches “depression”)
4. Keywords with * use wildcard regex (* → any non-whitespace)
5. Every match = one FINDING with a text snippet ±50 chars
6. Rule FAIL if any keyword hits · Rule PASS if zero hits
7. Overall PASS only if all rules pass (zero findings total)

Files to customise rules:

  • src/config/medexpress.rules.json — add/remove keywords, rules, categories
  • src/engine/medexpressKeywordEngine.js — matching algorithm (rarely need to change)

Phase 4 — BMI & vitals extraction

Separate from keyword rules. Parses the SCR section “Clinical Observations and Findings”.

MetricRow labels matched (case-insensitive)Output
Weight“body weight”, “o / e - weight”, or “weight”Latest by date → kg + date
Height“standing height”, “o / e - height”, or “height”Latest by date → value + date
BMI“body mass index”Latest by date → kg/m² + date

Values read from table.scr-table in that section. Multi-line rows use a spacer row + .scr-supporting-inset for the value.

File: src/engine/bmiExtractor.js

Note: Height is stored as the numeric value from SCR (e.g. 1.68 may display as “1.68 cm” — unit labelling in results is cosmetic).

Phase 5 — Results page display

Opens MedExpress keyword tool (keywords-tool.html + compiled/keywords-tool.js) — same React UI as MedExpress SCR extension.

SCR HTML is sent via renderSCRContent message after the tab loads. Patient header uses order-page userData.

Custom EDM UI (optional): src/ui/result.html — not used by default; customise MedExpress tool or fork compiled/ when ready.

What is NOT scanned today

  • Individual SCR sections (Allergies, Diagnoses, etc.) are not parsed into structured fields for results — only flattened text keyword search + BMI table
  • NHS number / address from SCR banner are not shown on results (order page name/DOB used instead)
  • Medications list on results page is empty unless you add a parser (placeholder in code)
  • NHS portal “Search this Summary Care Record” box is not used — extension searches offline in extracted HTML

How to customise (quick guide)

GoalEdit
Add/remove a keywordsrc/config/medexpress.rules.json
Add a new rule R007New object in same JSON with id, name, keywords, fail_if_found: true
Change order page selectorsmain-page.js
Change NHS org checkconfig.js → AUTH_TEXT / AUTH_ORG_CODE
Change results page looksrc/ui/result.html + result.css
Stricter matching (whole words only)src/engine/medexpressKeywordEngine.js → findKeywordMatches()

Rule JSON shape (for new rules)

{
  "id": "R007",
  "name": "My new rule",
  "category": "condition",
  "keywords": ["keyword1", "partial*match"],
  "fail_if_found": true,
  "weight": 1
}
